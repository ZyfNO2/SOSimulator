# AI 开发规则：React 拖拽卡牌网页小游戏

## 1. 总原则

- 先保证 MVP 可玩，再扩展内容。
- 不复刻现成游戏的文本、数值和专有世界观。
- 所有玩法规则优先数据驱动。
- 任何新规则都要能被测试。
- 不要把关键逻辑藏在 UI 组件里。

## 2. React 规则

- 优先使用函数组件。
- 游戏主状态使用 reducer/store 集中管理。
- 不要用多个 `useEffect` 串联推导游戏状态。
- 可从 state 派生的数据用 selector 计算。
- `useEffect` 只用于真实副作用，如计时器、存档、浏览器事件监听。
- effect 必须有清理逻辑，尤其是 interval、timeout、事件监听。

## 3. TypeScript/数据规则

- 所有核心数据结构必须有显式类型。
- 卡牌和配方逻辑使用 `id`、`type`、`tags`，不要使用中文显示名做判断。
- `CardDefinition` 是静态配置，`CardInstance` 是运行时状态。
- 配方匹配必须是纯函数。
- 新增卡牌、动作、事件时，优先改 `data/` 配置，不改组件。

## 4. 状态更新规则

- reducer 中不要直接修改原对象。
- 非法操作返回原状态，并让 UI 显示反馈。
- 动作运行中，输入卡牌必须被锁定或从可拖拽区移除。
- 动作结算必须一次性处理：消耗、产出、日志、flag、事件、结局。
- 结局触发后禁止启动新动作。

## 5. 拖拽规则

- 拖拽源只暴露 `cardId`。
- 放置目标只根据 `canPlaceCardInSlot` 判断合法性。
- UI 的合法/非法高亮不能替代 reducer 校验。
- 必须预留点击替代方案，不要让拖拽成为唯一可能交互。

## 6. 文案规则

- 玩家可见文案使用中文。
- 类型名、接口名、文件名使用英文。
- 日志文本要提供轻度提示，不要只写氛围句。
- 新增术语前先检查 `Glossary.md`。

## 7. 文件组织规则

- UI 组件放在 `components/`。
- 静态配置放在 `data/`。
- 纯规则、reducer、selectors 放在 `game/`。
- 测试放在项目约定测试目录；没有约定时使用 `src/tests/` 或 `__tests__/`。
- 不要为了一个 MVP 引入复杂状态库，除非项目已有并正在使用。

## 8. 测试规则

- 重构规则函数前，先补测试。
- 每个新增配方至少有一个匹配测试。
- 每个新增动作至少有一个开始/结算测试。
- 胜利路径必须有端到端测试或等价集成测试。
- UI 拖拽变更需要用 Playwright 实际验证。

## 9. 禁止事项

- 禁止在组件里写大段 `if/else` 决定配方产出。
- 禁止用卡牌中文名判断规则。
- 禁止让倒计时完成时自动悄悄吞掉输入而没有日志。
- 禁止让非法拖拽静默失败。
- 禁止在 MVP 阶段加入后端、登录、联网同步作为必要依赖。
- 禁止生成大量未被使用的抽象层。

## 10. 开发顺序建议

1. 建类型和初始数据。
2. 建 reducer 与规则函数。
3. 写规则单元测试。
4. 建基础 UI 布局。
5. 接入拖拽。
6. 接入动作倒计时和结算。
7. 接入事件和胜利结局。
8. 用 Playwright 验证完整路径。

## 11. 完成定义

一次实现任务只有满足以下条件才算完成：

- 代码可运行。
- 关键规则有测试。
- UI 交互实际验证过。
- 文档中新增或改变的约定已同步。
- 没有引入与 MVP 无关的大型依赖。
