# SOSimulator 现阶段玩法改造工作流

## 1. 文档目的

这份文档只说明“当前代码已经落地到哪一步”和“接下来还能往哪一步走”，不再把未来设计稿写得像已经实现。

尤其需要明确：

- 当前代码已经不是密教模拟器式动作框
- 当前代码已经进入蓝伞主线版桌面原型
- 但“按天推进 / 第四天关店 / 打工线中止”仍属于故事大纲层设想，尚未进入运行逻辑

---

## 2. 当前代码已经实现的方向

## 2.1 开场

- 开始界面先遮住桌面
- 点击开始后，初始 5 张牌依次显现：
  - `精力`
  - `时间`
  - `春日0`
  - `商店`
  - `天气（晴）`

## 2.2 主循环

当前主循环已经固定为：

1. 从母牌中分离 `精力 / 时间`
2. 在桌面上把卡牌堆叠成父子链
3. 自动匹配合成规则
4. 等待倒计时产出
5. 继续推进蓝伞主线与天气压力

这意味着玩家当前体验到的是一个“连续桌面节奏”，而不是“按天结算”的离散循环。

## 2.3 资源压力

当前资源压力已经由母牌系统实现：

- `精力 / 时间` 固定左下角
- 总量上限为 `2`
- 只有母牌自动恢复
- 资源分离、回收、回充遮罩都已落地

## 2.4 天气压力

天气现在已经不再随剧情节点手动升级，而是：

- 首次发现任意阶段的 `蓝色雨伞` 后启动独立计时
- 每 `3` 分钟升级一级
- 到 `天气（特大暴雨）` 后，再过 `3` 分钟强制进入 `虚构失主出现`

这一点已经是代码中的真实规则，不再只是设计方向。

---

## 3. 当前代码尚未实现的叙事层

虽然故事大纲里有很强的“打工 / 第几天 / 第四天关店”结构，但当前原型还没有下列系统：

- `第 1 天 / 第 2 天 / 第 3 天 / 第 4 天`
- 店铺营业状态
- 第四天店铺关闭
- 每日刷新结算
- 打工路线被中断的剧情事件

换句话说：

- 故事大纲里“为了团费打工”仍然是叙事依据
- 代码里真正落地的“现实线”目前只压缩成了 `商店 + 精力 -> 金钱 -> SOS团活动材料`

---

## 4. 当前可视表达已经承担的叙事功能

当前代码不是靠独立文本框讲故事，而是靠这些系统共同表达：

- `CardKind.json` 中的卡牌简介
- `EventCardDetail` 里的详情弹窗
- `Back_text.json` 的背景滑动文字
- 天气牌的倒计时遮罩
- 主线卡收纳栏
- 结局牌的清场效果

所以现阶段的设计工作流，重点已经不是“先加剧情框”，而是：

1. 先保证卡牌命名和产出链准确
2. 再保证详情文案、背景文案和天气节奏一致
3. 最后再决定是否值得补更重的章节系统

---

## 5. 当前最稳定的设计真值来源

如果文档之间发生冲突，应以当前这些内容为准：

1. `src/data/CardOutput.json`
2. `src/data/CardKind.json`
3. `src/App.tsx` 中的天气与母牌逻辑
4. [卡牌合成表.md](D:\github\凉宫春日黑客松\SOSimulator\doc\卡牌合成表.md)

原因很简单：这些地方已经决定了当前玩家真正能玩到什么。

## 6. 当前原型适合继续加强的部分

如果只顺着当前代码往前走，最合适继续补的是：

### 6.1 文案层

- 为关键卡手写详情提示
- 继续增强背景文字
- 区分不同结局线的详情语气

### 6.2 节奏层

- 增加更多“保留输入牌”的产出动画表现
- 为临时状态卡加更明确的视觉倒计时
- 加重天气升级带来的压迫感

### 6.3 规则层

- 引入人物冷却
- 引入干扰事件
- 引入结局前的额外要求

---

## 7. 暂时不建议马上补的部分

以下内容虽然在大纲里很好，但现在直接做会明显变成另一轮系统工程：

- 真正的按天推进
- 第四天关店事件
- 独立日志面板
- 动作框 / 槽位式系统回归
- 完整多角色卡池扩张

这些内容一旦开做，就不再只是“调规则”，而是会牵动状态结构和 UI 结构。

---

## 8. 当前工作流建议

在当前代码基础上，更顺的推进顺序是：

1. 先保持 [卡牌合成表.md](D:\github\凉宫春日黑客松\SOSimulator\doc\卡牌合成表.md) 与代码完全同步
2. 再同步 `CardKind.json` 的卡片简介和 `Event_Card.json` 的提示口吻
3. 然后补 `Back_text.json` 与详情文案的叙事一致性
4. 最后才决定要不要上“第几天 / 关店 / 干扰事件”

也就是说，当前最稳的策略不是继续扩系统，而是先把这版桌面原型的叙事表达打磨完整。
